# Security

## Defense in Depth

This deployment implements multiple security layers:

### 1. Network Security

**VPC Isolation**
- EKS nodes and RDS instances run in private subnets (no direct internet access)
- Public subnets only contain ALB and NAT Gateway
- VPC Flow Logs enabled for network traffic auditing

**Security Groups**
- RDS: Only accepts connections from EKS node security group on port 5432
- EKS nodes: Controlled ingress from cluster control plane
- No overly permissive 0.0.0.0/0 ingress rules on compute resources

**Kubernetes NetworkPolicies**
- Default-deny policy blocks all traffic by default
- Explicit allow rules for:
  - Kong ingress (port 8000 from any, for ALB health checks)
  - Inter-service communication within the supabase namespace
  - DNS egress (port 53 UDP/TCP)
  - RDS egress (port 5432 to VPC CIDR)
  - S3/STS egress (port 443 for IRSA + storage)

### 2. Data Encryption

**At Rest**
- RDS: AES-256 encryption enabled by default
- S3: Server-side encryption with AES-256 (SSE-S3)
- Secrets Manager: AWS-managed encryption

**In Transit**
- TLS termination at ALB (cert-manager with Let's Encrypt)
- RDS connections use SSL (DB_SSL=require in service configs)
- S3 bucket policy enforces SSL-only access (denies non-HTTPS requests)

### 3. Identity and Access Management

**IRSA (IAM Roles for Service Accounts)**
- External Secrets Operator: `secretsmanager:GetSecretValue`, `secretsmanager:DescribeSecret`
- Supabase Storage: Scoped S3 access (Get/Put/Delete/List on specific bucket)
- Karpenter: EC2 management for autoscaling

Each role uses OIDC-based trust policies with condition keys that restrict access to specific Kubernetes service accounts in specific namespaces.

**No Static Credentials**
- All AWS access uses short-lived STS tokens via IRSA
- No AWS access keys stored in Kubernetes Secrets or environment variables

### 4. Secret Management

**Generation**
- Passwords generated by Terraform's `random_password` provider
- JWT secret: 64 characters
- Database password: 32 characters with special characters
- Service keys: 64 characters

**Storage**
- Single JSON secret in AWS Secrets Manager
- External Secrets Operator syncs to Kubernetes Secrets
- Hourly refresh interval ensures secrets stay current

**Rotation**
- `scripts/rotate-secrets.sh` automates rotation:
  1. Terraform regenerates passwords and updates Secrets Manager
  2. ESO annotation triggers immediate re-sync
  3. Rolling restart applies new secrets to pods

### 5. S3 Bucket Security

- All public access blocked (four-way block)
- Bucket policy denies non-SSL requests
- Versioning enabled (protects against accidental deletion)
- Lifecycle policy cleans incomplete multipart uploads

### 6. Database Security

- Multi-AZ for availability (not security, but reduces blast radius of AZ-level incidents)
- Deletion protection enabled by default
- 7-day automated backup retention
- Performance Insights for anomaly detection
- Custom parameter group (no default PG configs)

## Compliance Considerations

| Control | Implementation |
|---------|----------------|
| Encryption at rest | RDS (AES-256), S3 (SSE-S3), Secrets Manager |
| Encryption in transit | TLS at ALB, SSL to RDS, HTTPS-only S3 |
| Least privilege | IRSA with scoped policies |
| Network segmentation | Private subnets, security groups, NetworkPolicies |
| Audit logging | VPC Flow Logs, EKS control plane logs |
| Secret management | AWS Secrets Manager + ESO (no plaintext secrets in Git) |
| Backup and recovery | RDS automated backups, S3 versioning |

## Known Limitations

1. **Single NAT Gateway**: If the AZ hosting the NAT goes down, outbound internet access is disrupted
2. **No WAF**: ALB does not have WAF rules for DDoS/bot protection (future improvement)
3. **No pod security standards**: PodSecurityPolicy/PodSecurityStandard not configured
4. **Certificate rotation**: Manual cert-manager setup required post-deployment
